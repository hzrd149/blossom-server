## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> PUT /upload | Untrusted request body and headers; X-SHA-256 is client-declared hash |
| client -> HEAD /upload | Untrusted preflight headers (X-Content-Length, X-Content-Type, X-SHA-256) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Spoofing | upload.ts HEAD handler | accept | HEAD /upload preflight is informational; 204 vs 200 does not grant access. Auth is enforced when required. |
| T-01-05 | Information Disclosure | upload.ts 409 response | accept | 409 response includes declared vs computed hash in X-Reason. This is intentional per BUD spec — the client already knows the declared hash (they sent it), and the computed hash is the blob's content-addressed identity (public by design). |
| T-01-06 | Denial of Service | upload.ts 201 path | accept | No change to upload admission logic — 201 vs 200 is purely the response code. Existing size limits, auth, and pool capacity checks remain. |
</threat_model>
